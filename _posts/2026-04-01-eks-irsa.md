## IRSA란?

IRSA(IAM Roles for Service Accounts)는 Kubernetes의 ServiceAccount에 AWS IAM Role을 연결하는 메커니즘이다. Pod가 AWS 리소스에 접근할 때 Node에 붙은 IAM Role을 공유하는 대신, Pod 단위로 최소 권한을 부여할 수 있다.

## IRSA가 없으면?

### 1. Node IAM Role 공유

```
Node (IAM Role: S3 + DynamoDB + SQS 전부 허용)
├── Pod A (S3만 필요)        → 전부 접근 가능
├── Pod B (DynamoDB만 필요)  → 전부 접근 가능
└── Pod C (SQS만 필요)       → 전부 접근 가능
```

한 Pod가 탈취되면 해당 Node의 모든 AWS 권한이 노출된다.

### 2. Access Key를 Secret에 저장

키 유출 위험이 있고, 로테이션을 수동으로 관리해야 한다.

**IRSA는 이 두 가지 문제를 모두 해결한다.**

## 동작 원리

```
① Pod 생성 → ServiceAccount의 IAM Role ARN 확인, OIDC JWT 토큰을 Pod에 마운트
    │
② Pod 내 AWS SDK → 마운트된 JWT 토큰으로 STS AssumeRoleWithWebIdentity 호출
    │
③ AWS STS → OIDC Provider에 토큰 검증, Namespace/ServiceAccount 조건 확인
    │
④ 검증 통과 → 임시 자격 증명 발급 → Pod가 AWS 리소스 접근
```

**정적 키 없이 OIDC 토큰 → STS 임시 자격 증명** 흐름으로 동작한다.

## Terraform 설정

### 1. OIDC Provider 등록

```hcl
data "tls_certificate" "eks" {
  url = aws_eks_cluster.main.identity[0].oidc[0].issuer
}

resource "aws_iam_openid_connect_provider" "eks" {
  client_id_list  = ["sts.amazonaws.com"]
  thumbprint_list = [data.tls_certificate.eks.certificates[0].sha1_fingerprint]
  url             = aws_eks_cluster.main.identity[0].oidc[0].issuer
}
```

### 2. IAM Role + Trust Policy

```hcl
data "aws_iam_policy_document" "my_app_assume" {
  statement {
    actions = ["sts:AssumeRoleWithWebIdentity"]
    effect  = "Allow"

    principals {
      type        = "Federated"
      identifiers = [aws_iam_openid_connect_provider.eks.arn]
    }

    condition {
      test     = "StringEquals"
      variable = "${replace(aws_eks_cluster.main.identity[0].oidc[0].issuer, "https://", "")}:sub"
      values   = ["system:serviceaccount:<namespace>:<service-account>"]
    }

    condition {
      test     = "StringEquals"
      variable = "${replace(aws_eks_cluster.main.identity[0].oidc[0].issuer, "https://", "")}:aud"
      values   = ["sts.amazonaws.com"]
    }
  }
}

resource "aws_iam_role" "my_app" {
  name               = "my-app-eks"
  assume_role_policy = data.aws_iam_policy_document.my_app_assume.json
}
```

`:sub` 조건이 핵심이다. 특정 Namespace의 특정 ServiceAccount만 이 Role을 assume할 수 있도록 제한한다.

### 3. 권한 정책 연결

```hcl
resource "aws_iam_role_policy" "my_app" {
  role   = aws_iam_role.my_app.id
  policy = data.aws_iam_policy_document.my_app_policy.json
}
```

서비스마다 필요한 권한만 부여한다.

### 4. Helm Chart에서 ServiceAccount 연결

```hcl
resource "helm_release" "my_app" {
  # ...
  set {
    name  = "serviceAccount.annotations.eks\\.amazonaws\\.com/role-arn"
    value = var.my_app_role_arn
  }
}
```

이 annotation이 붙으면 EKS Mutating Webhook이 Pod에 환경변수(`AWS_ROLE_ARN`, `AWS_WEB_IDENTITY_TOKEN_FILE`)와 OIDC 토큰 볼륨을 자동 주입한다. AWS SDK가 이를 인식하므로 별도 credential 설정이 필요 없다.

### 5. Output으로 Role ARN 전달

EKS 모듈에서 IRSA Role을 생성하고, Helm 배포 모듈로 ARN을 넘기는 구조가 깔끔하다.

```hcl
output "my_app_role_arn" {
  value = aws_iam_role.my_app.arn
}
```

## 검증

```bash
kubectl exec <pod-name> -- aws sts get-caller-identity
```

Node Role이 아닌 IRSA로 지정한 Role로 인증되는 것을 확인할 수 있다.

## 정리

| 항목 | Node IAM Role | Access Key | IRSA |
|---|---|---|---|
| 권한 범위 | Node 전체 | Pod 단위 | Pod 단위 |
| 키 관리 | 불필요 | 수동 로테이션 | 불필요 (임시 자격 증명) |
| 최소 권한 | 어려움 | 가능 | 가능 |
| 보안 | 과도한 권한 공유 | 키 유출 위험 | OIDC 기반 안전 |
