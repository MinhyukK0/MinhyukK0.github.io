## Spark on Kubernetes

Spark 2.3부터 Kubernetes를 네이티브 클러스터 매니저로 지원한다. YARN이나 Mesos 없이 K8s 위에서 직접 Spark 애플리케이션을 실행할 수 있다.

K8s에서 Spark를 실행하는 방법은 크게 세 가지다.

| 방식 | 설명 |
|---|---|
| `spark-submit` | Spark 기본 제공 CLI로 직접 K8s에 제출 |
| **Spark Operator** | CRD 기반으로 Spark 앱을 K8s 리소스로 관리 |
| EMR on EKS | AWS 관리형, 내부적으로 spark-submit을 래핑 |

이 글에서는 Spark Operator 방식을 다룬다.

## spark-submit의 한계

`spark-submit --master k8s://...`로 직접 제출할 수 있지만 몇 가지 불편함이 있다.

- 제출 자체가 명령어 기반이라 **선언적 관리가 안 된다**
- 앱의 상태를 K8s 리소스로 추적할 수 없다
- 재시도, 스케줄링 등의 정책을 직접 구현해야 한다
- GitOps나 Helm 기반 배포 파이프라인에 녹이기 어렵다

## Spark Operator

[Spark Operator](https://github.com/kubeflow/spark-operator)는 Kubernetes Operator 패턴으로 Spark 애플리케이션의 라이프사이클을 관리한다.

### 구조

```
Spark Operator (Controller)
    │
    ├── SparkApplication CRD 감시
    │
    ├── 감지 시 spark-submit 실행
    │       │
    │       ├── Driver Pod 생성
    │       │       │
    │       │       └── Executor Pod(s) 생성
    │       │
    │       └── 상태를 CRD Status에 반영
    │
    └── 완료/실패 시 재시도 정책 적용
```

핵심은 **CRD로 선언하면 Operator가 spark-submit을 대신 실행**한다는 것이다. 사용자는 YAML만 관리하면 된다.

### SparkApplication 예시

```yaml
apiVersion: sparkoperator.k8s.io/v1beta2
kind: SparkApplication
metadata:
  name: my-spark-job
  namespace: spark
spec:
  type: Python
  mode: cluster
  image: <spark-image>
  mainApplicationFile: s3a://my-bucket/jobs/main.py
  sparkVersion: "3.5"

  driver:
    cores: 1
    memory: "2g"
    serviceAccount: spark

  executor:
    cores: 2
    instances: 3
    memory: "4g"

  restartPolicy:
    type: OnFailure
    onFailureRetries: 3
```

이것만으로 Driver 1개 + Executor 3개가 뜨고, 실패 시 3번까지 재시도한다.

### Driver와 Executor

```
┌─────────────────────────────────────────┐
│ Kubernetes Cluster                      │
│                                         │
│   ┌──────────┐                          │
│   │  Driver   │ ── 작업 분배 ──┬────┐   │
│   │   Pod     │               │    │   │
│   └──────────┘               ▼    ▼   │
│                         ┌──────┐┌──────┐│
│                         │Exec 1││Exec 2││
│                         │ Pod  ││ Pod  ││
│                         └──────┘└──────┘│
│                               ▼         │
│                          S3 / HDFS      │
└─────────────────────────────────────────┘
```

- **Driver**: 작업을 분할하고 Executor에 분배, 결과를 수집
- **Executor**: 실제 데이터 처리 수행, 완료 후 자동 종료

Executor Pod는 작업이 끝나면 사라진다. K8s 리소스를 점유하지 않는다.

### ScheduledSparkApplication

주기적으로 실행해야 하는 배치 작업은 `ScheduledSparkApplication`을 사용한다.

```yaml
apiVersion: sparkoperator.k8s.io/v1beta2
kind: ScheduledSparkApplication
metadata:
  name: my-daily-job
  namespace: spark
spec:
  schedule: "0 2 * * *"
  concurrencyPolicy: Forbid
  template:
    # SparkApplication spec과 동일
```

CronJob처럼 cron 표현식으로 스케줄링할 수 있다.

## Spark Operator를 쓰는 이유

| 항목 | spark-submit | Spark Operator |
|---|---|---|
| 관리 방식 | 명령어 | CRD (선언적) |
| 상태 추적 | 별도 구현 | `kubectl get sparkapplication` |
| 재시도 | 수동 | `restartPolicy` 설정 |
| 스케줄링 | 외부 도구 필요 | `ScheduledSparkApplication` |
| Helm/GitOps | 어려움 | 자연스러움 |

Kubernetes 위에서 Spark를 운영한다면 Spark Operator가 표준적인 선택이다.
