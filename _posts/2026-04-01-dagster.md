## Dagster란?

Dagster는 데이터 파이프라인을 **Asset 중심**으로 관리하는 오케스트레이션 프레임워크다. 기존 오케스트레이터(Airflow 등)가 "작업을 어떤 순서로 실행할까"에 집중한다면, Dagster는 "어떤 데이터를 만들어야 하는가"에 집중한다.

## Task 기반 vs Asset 기반

### Airflow (Task 기반)

```
extract → transform → load
```

DAG에 Task의 실행 순서를 정의한다. Task 간 의존성은 있지만, 각 Task가 어떤 데이터를 만드는지는 코드를 봐야 안다.

### Dagster (Asset 기반)

```
raw_data (Source)
    │
    ▼
cleaned_data (Asset)
    │
    ▼
report_table (Asset)
```

Asset을 선언하면 의존성에 따라 실행 순서가 자동으로 결정된다. 어떤 데이터가 있고, 언제 마지막으로 갱신됐는지 UI에서 바로 확인할 수 있다.

### 코드 비교

```python
# Airflow - Task 정의
@task
def extract():
    ...

@task
def transform(data):
    ...

extract() >> transform()
```

```python
# Dagster - Asset 정의
@asset
def raw_data():
    return fetch_from_source()

@asset
def cleaned_data(raw_data):
    return clean(raw_data)
```

Dagster는 함수의 파라미터가 곧 의존성이다. `cleaned_data`는 `raw_data`에 의존한다는 것을 프레임워크가 자동으로 인식한다.

## 핵심 개념

### Asset

파이프라인이 생산하는 데이터 단위다. 테이블, 파일, ML 모델 등 무엇이든 Asset으로 정의할 수 있다. Asset 간 의존 관계를 선언하면 Dagster가 실행 순서와 갱신 전략을 관리한다.

### Resource

Asset이 사용하는 외부 시스템과의 연결이다. DB 커넥션, S3 클라이언트, API 클라이언트 등을 Resource로 정의하면 환경별로(dev/prod) 교체가 쉽다.

```python
@asset
def my_table(database: Resource[DatabaseClient]):
    database.execute("SELECT ...")
```

### Partition

데이터를 시간/키 기준으로 분할 처리할 수 있다. 특정 날짜만 재처리(backfill)하는 것이 간단하다.

```python
@asset(partitions_def=DailyPartitionsDefinition(start_date="2026-01-01"))
def daily_report(context):
    date = context.partition_key  # "2026-04-01"
    ...
```

### Schedule & Sensor

- **Schedule**: cron 기반 주기 실행
- **Sensor**: 외부 이벤트(S3 파일 도착, DB 변경 등)를 감지해서 실행

## Dagster on Kubernetes 아키텍처

**Kubernetes Cluster 내 구성:**

- **Webserver (UI)** — Asset 상태 확인, 수동 실행
- **Daemon** — Schedule/Sensor 평가, Run 큐 관리
- **Database (PostgreSQL)** — Run 이력, Asset 메타데이터 저장
- **Run Pod (K8sRunLauncher)** — Daemon이 트리거하면 생성, 실제 파이프라인 실행 후 종료

### 구성 요소

| 컴포넌트 | 역할 |
|---|---|
| **Webserver** | UI 제공, Asset 상태 확인, 수동 실행 |
| **Daemon** | Schedule/Sensor 평가, Run 큐 관리 |
| **Database** | Run 이력, Asset 메타데이터 저장 |
| **Run Pod** | 실제 파이프라인 실행, 완료 후 종료 |

### K8s Run Launcher

Dagster는 `K8sRunLauncher`를 통해 **Run마다 별도의 Pod**를 생성한다. Run이 끝나면 Pod는 종료되므로 리소스를 점유하지 않는다.

Helm으로 배포 시 ServiceAccount에 IRSA를 연결하면 Run Pod가 AWS 리소스에 접근할 수 있다.

```yaml
# Helm values 예시
runLauncher:
  type: K8sRunLauncher
  config:
    serviceAccount: dagster
```

## Airflow와 비교

| 항목 | Airflow | Dagster |
|---|---|---|
| 핵심 단위 | Task (작업) | Asset (데이터) |
| 의존성 | 명시적 선언 (`>>`) | 함수 파라미터로 자동 |
| 데이터 가시성 | 별도 구현 필요 | Asset UI 기본 제공 |
| 환경 분리 | 설정 복잡 | Resource 교체로 간단 |
| Backfill | 날짜 범위 지정 | Partition 단위로 선택 |
| K8s 실행 | KubernetesExecutor | K8sRunLauncher |

Airflow는 성숙한 생태계와 넓은 커뮤니티가 강점이고, Dagster는 Asset 중심의 데이터 관리와 개발 경험이 강점이다.
